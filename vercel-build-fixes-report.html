Problems and Easy Fixes

1. Vercel failed because of js-cookie

Problem: Vercel showed: Could not find a declaration file for module 'js-cookie'.

File: src/features/account/server/account.actions.ts

Why: The file used js-cookie, but the app stores the token as an httpOnly cookie. Browser tools like js-cookie cannot read httpOnly cookies.

Solution: Make the function a server action and read the cookie using Next.js cookies().

  1. Add "use server" at the top of the file.
  2. Remove import Cookies from 'js-cookie'.
  3. Add import { cookies } from "next/headers".
  4. Read the token with const token = cookieStore.get("token")?.value.

Easy idea: if the cookie is protected on the server, read it on the server.

2. Signup gender validation used old Zod syntax

Problem: TypeScript rejected errorMap inside z.enum.

File: src/features/auth/schemas/signup.schema.ts

Why: The project uses Zod 4, but the code used older Zod syntax.

Solution: Use the simpler Zod 4 enum message syntax:

gender: z.enum(["male", "female"], "Please select your gender")

Easy idea: when a library version changes, some old options may stop working.

3. Cart screen imported something that was not exported

Problem: The cart screen imported cartSlice, but cartSlice was not exported.

File: src/features/cart/screens/cart.screen.tsx

Why: TypeScript only allows importing names that the other file exports.

Solution: Remove the unused import:

import { cartSlice } from "../store/cart.slice";

Easy idea: if you do not use an import, delete it.

4. Cart product type was missing quantity

Problem: The cart page used item.product.quantity, but the type did not include quantity.

File: src/features/cart/store/cart.slice.ts

Why: TypeScript must know every property you use.

Solution: Add quantity: number to CartProductDetails.

Easy idea: if the UI uses a field from the API, add that field to the TypeScript interface.

5. Orders function expected an id but did not need it

Problem: The function was written as getUserOrder(id: string), but the page called getUserOrder().

File: src/features/orders/server/orders.actions.ts

Why: The function already gets the user id from the token, so the parameter was unnecessary.

Solution: Remove the id parameter and return Promise<Order[]>.

Easy idea: the function definition and the function call must match.

6. Product type was missing priceAfterDiscount

Problem: Product cards used priceAfterDiscount, but the shared Product type did not include it.

File: src/features/products/types/products.types.ts

Why: Some products can have discounts, so this field may exist in the API data.

Solution: Add an optional field:

priceAfterDiscount?: number

Easy idea: use ? when a field may or may not exist.

7. Product details had a duplicate local Product type

Problem: ProductInfo.tsx had its own product type, and one property was typed with a spelling mistake.

File: src/features/products/components/ProductDetails/ProductInfo.tsx

Why: Copying the same type in many places makes mistakes easy.

Solution: Delete the local type and import the shared Product type.

import type { Product } from "../../types/products.types";

Easy idea: use one shared type instead of repeating the same shape in many files.

8. Providers.tsx used any and ignored preloadedState

Problem: preloadedState?: any caused an ESLint error, and preloadedState was passed but not used.

File: src/components/providers/Providers.tsx

Why: any removes TypeScript safety. Also, the server sent auth/cart data to Redux, but Redux ignored it.

Solution: Type it as Partial<RootState> and use it when creating the Redux store.

  1. Create the store with makeStore(preloadedState).
  2. Use useState(() => makeStore(preloadedState)) so the store is created once.
  3. Pass the created store to <Provider store={store}>.

Easy idea: if a value is passed to a component, either use it or remove it.

9. Redux store needed to support initial server data

Problem: The store was a fixed exported store and could not easily receive initial data from layout.tsx.

File: src/features/auth/store/store.ts

Why: Next.js can load auth/cart data on the server first, then send it to the client. Redux needs a way to start with that data.

Solution: Add a makeStore(preloadedState) function and use combineReducers.

Easy idea: server data should become the starting state for the client store.

Final Check

The project production build passed after the fixes.

The remaining warning about multiple lockfiles is not the same as an error. It means there is another package-lock.json outside the project folder on the local computer.

Simple Rule for Future Errors

  1. Read the first error message carefully.
  2. Open the file and line number from the error.
  3. Ask: is this a missing type, wrong import, wrong function call, or old library syntax?
  4. Fix one error at a time.
  5. Build again and repeat until no errors remain.
